## 3) Базовая проверка перед деплоем
Выполнять из корня проекта:

```bash
git status --short
git remote -v
git rev-parse --abbrev-ref HEAD
npm run build
```

Критерии:
- нет неожиданных файлов в изменениях;
- `remote` настроен (если деплой через push);
- сборка `npm run build` успешна.

## 4) Основной путь деплоя (рекомендуется): Git push -> Vercel Auto Deploy
1. Закоммитить изменения:
```bash
git add <files>
git commit -m "fix: <short description>"
```
2. Запушить в production-ветку (обычно `main`):
```bash
git push origin main
```
3. Дождаться автодеплоя в Vercel.

## 5) Альтернативы деплоя
Использовать, если push-процесс недоступен.

### 5.1 Прямой CLI деплой в Vercel
```bash
npx --yes vercel --prod --yes
```

### 5.2 Deploy Hook
Если настроен Deploy Hook URL, запускать деплой через hook (без push).

## 6) Smoke-check после каждого деплоя
Минимум:
- `/gazeta`
- `/gazeta/tourism`
- `/tourism-service`
- `/drone-service`

Проверить:
- нет визуального перекрытия fixed/sticky блоков;
- основные CTA и ссылки кликабельны;
- консоль без критичных ошибок;
- страницы открываются без 404/500.

## 7) Откат (rollback)
Есть 2 стандартных варианта.

### Вариант A: быстрый откат в Vercel (предпочтительно при срочном инциденте)
1. Открыть `Deployments` в Vercel проекта.
2. Найти последний стабильный деплой.
3. Нажать `Promote to Production` (или аналог rollback/promote).
4. Пройти smoke-check из раздела 6.

### Вариант B: откат через Git (долговременная фиксация)
1. Найти проблемный коммит.
2. Сделать `revert` (не reset):
```bash
git revert <bad_commit_sha>
git push origin main
```
3. Дождаться автодеплоя.
4. Пройти smoke-check.

Запрещено для рабочих откатов:
- `git reset --hard` по общей ветке;
- force-push в production-ветку без отдельного согласования.

## 8) Инцидент-процедура (если “сайт сломался”)
1. Зафиксировать симптомы: URL, время, скрин, что именно не работает.
2. Немедленно откатить (раздел 7, вариант A).
3. После восстановления продакшена разобрать причину в отдельной ветке.
4. Добавить короткий постмортем в `CONTEXT_NEXT_CHAT.md`:
   - что сломалось;
   - какой коммит/деплой виноват;
   - как откатили;
   - как предотвращаем повтор.

## 9) Роли документов
- `CONTEXT_NEXT_CHAT.md`: оперативный контекст, что сделано/что болит сейчас.
- `DEPLOYMENT.md`: постоянный регламент релизов и откатов.
